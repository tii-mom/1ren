# 后端账本与用户身份设计 (Backend Ledger & Identity Design)

本设计文档旨在为 1人算力公司 (1ren) 规划下一阶段的最小后端账本与用户身份模型。通过将核心资产与设备管理逻辑转移至受控的后端服务，以保障状态的一致性、防篡改性，并为多端同步及未来的生产环境奠定技术基础。

---

## 1. 为什么当前 localStorage 只能用于 V1 Demo

当前版本的系统完全依赖前端 `localStorage` 存储所有核心数据（用户算力、USDT 余额、已购设备、企业代币发行和订单记录）。这种架构仅适用于早期 Demo 和演示验证，无法用于真实的生产运行，原因如下：
1. **安全性缺失**：用户可以通过浏览器控制台或直接修改本地存储任意修改自身的 USDT 余额、AI Token 数量和设备属性。
2. **状态无法跨设备/多端同步**：一旦清除浏览器缓存或更换设备，用户所有的资产和部署记录将彻底丢失。
3. **缺乏一致性保障**：在涉及交易（如购买设备、出售 AI Token、R1 交易、公司代币支持）时，前端无法确保资产扣减与记录生成的原子性（Atomicity），极易因刷新或崩溃导致数据处于半更新状态。
4. **防作弊及安全防御能力差**：设备挖矿产出完全由前端 ticking 计算决定，攻击者只需伪造或加速时间周期即可无限制刷取 AI Token 利润。

---

## 2. 后端账本的目标 (Backend Ledger Objectives)

引入后端账本的目标是为了建立 **唯一可信的状态中心 (Single Source of Truth)**，将数据读写、业务校验及状态流转的生命周期完全收敛在后端：
- **可信任的设备周期与收益结算**：矿机运行时长、维护损耗及收益计算均由后端 cron 定期结算，前端仅负责同步和展示渲染。
- **账目原子性与流水追踪**：采用类似复式记账法或结构化账户流水的形式记录每一次资产变动，每一笔收支都必须有合法的业务实体（设备订单、交易单等）进行强绑定归因。
- **跨平台访问与高可用**：通过基于 Session 身份的后端存储，使用户可以随时跨多端（H5、微信小程序、Telegram Mini App 等）同步资产状态。

---

## 3. 用户身份模型 (User Identity Model)

本系统目前不需要对接真实的 TON 钱包或复杂的 OAuth 第三方，但也必须支持灵活的升级扩展路径。我们设计一个**最小化可升级的身份模型**：
- **游客身份/会话绑定**：初始访问时为用户生成一个唯一的 `session_token`，自动与一个匿名的用户 ID 关联，并在本地记录一个临时游客标识。
- **TG Mini App 握手 fallback**：允许接收 Telegram 初始化参数 (`initData`)。若从 TG 载入，直接解析提取 Telegram User ID 进行用户表的静默注册绑定与鉴权。
- **Wallet 备用占位**：用户表包含一个备用的 `wallet_address` 字段，可随时允许用户进行非强绑定形式的 Web3 钱包关联。

---

## 4. 资产账本模型 (Asset Ledger Model)

所有资产的转移（USDT, R1 Token, AI Token, 算力晶体等）必须通过资产账本表统一扣减和增加，杜绝字段的任意覆写。
- **动账明细流水 (Ledger Entries)**：
  - 核心采用 `ledger_entries` 流水记录。
  - 包含发生资产类型 (`asset_type`)、变化数值 (`amount`)、方向 (`debit` / `credit` 或 `direction`)、关联业务类型 (`action`)。
  - 用户的账户余额 (`asset_accounts`) 相当于对明细流水的快速聚合缓存（由行锁保护，防止并发动账冲突）。

---

## 5. 设备订单模型 (Device Order Model)

- 用户在商城购买（租赁）设备时，不再由前端生成设备对象。
- 前端提交商品 ID $\rightarrow$ 后端判定余额是否充足 $\rightarrow$ 后端扣除模拟 USDT 余额 $\rightarrow$ 后端向 `device_orders` 插入记录并生成一条运行中的设备记录 `devices`。
- **设备运行状态由后端判定**：后端根据当前时间与合同起止时间判定设备是否已过期；通过记录维护操作来流转设备效率等级。

---

## 6. 核心业务流水设计

### 6.1 AI Token 产出记录 (Mining Records)
- 由后端服务负责每隔一个结算周期（如每小时或每日）检查所有运行中的设备，并计算累计产出的 AI Token。
- 生成对应的 `mining_records`，并同时对用户的 AI Token 资产账户加额。

### 6.2 R1 交易记录 (R1 Spot Orders & Trades)
- R1/USDT 的极速买入和卖出交易均需经过后端的统一对账。
- 每次买卖均会在 `ledger_entries` 中同步生成一对一的“R1 扣减-USDT 增加”或“USDT 扣减-R1 增加”的流水。

### 6.3 USDT Mock Balance 记录
- USDT 目前完全为系统预设的模拟测试代币，所有测试金的添加（测试额度充值包）和消费都需要在后端 `ledger_entries` 留下业务动账印记。

### 6.4 公司 Token 发行记录与支持记录
- 用户在发行中心提交影子代币数据，后端在校验发行条件（设备数量、累计产出、R1 持有）后，向 `company_tokens` 表写入发行记录，状态为筹集中 (`launching`)。
- 当有其他用户（或当前用户）投入 USDT 进行模拟支持时，在 `company_token_supports` 中写入支持流水，并累加代币筹集进度；若满 100%，挂牌上市。

### 6.5 邀请归因记录 (Referrals)
- 用户账号在初次创建时可选绑定推荐码。
- 后端维护一条 `referrals` 邀请树链。
- 团队加成算力等逻辑由后端基于层级归因网络计算产生，防止前端脚本伪造关系链。

## 7. 风险边界 (Risk Boundaries)

由于该项目定位于“演示模拟体验与算力公司成长模拟”沙盒系统，系统架构在设计和对外宣称上必须严守以下安全风险边界：
1. **不承诺未来实际收益 (No Yield Promise)**：系统内的所有设备运行、AI Token 产出及 R1 权益回报均为沙盒模拟数据，不代表真实世界下的任何投资收益承诺。
2. **不做真实的金融资产清算 (No Real Financial Settlement)**：系统内的 USDT 及支付行为纯属演示沙盒，后端仅保存记录并支持模拟代币增加，不提供与法币或真实加密货币进行资金清算的兑换服务。
3. **不做强制性的链上映射承诺 (No Jetton / Chain Mapping Commitments)**：代币发行、挂牌或权益交换机制均在后端中心化数据库（Ledger）运行。系统不预设任何强制或无条件的智能合约映射、TON 链 Jetton 发行或 DEX 流通的法律承诺。
